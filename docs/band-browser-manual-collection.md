# BAND 브라우저 직접 수집 방식

수집일 기준: 2026-05-04

이 문서는 BUSHIRI에 BAND 가격 게시글을 넣을 때, 공식 API가 아니라 브라우저에서 게시글을 직접 열어 사람이 검수하듯 수집하는 절차를 정리한다. 참조은수산처럼 `https://www.band.us/page/...` 형태의 Page 게시글은 정적 HTML만으로 전체 본문이 노출되지 않으므로, 브라우저에서 로그인된 상태로 게시글 상세를 직접 열어 확인하는 방식을 기본으로 한다.

## 대상 URL

- 일반 밴드 게시글: `https://www.band.us/band/{bandId}/post/{postId}`
- BAND Page 게시글: `https://www.band.us/page/{pageId}/post/{postId}`
- BAND Page 목록: `https://www.band.us/page/{pageId}/post`

목록 화면만으로는 가격표가 누락될 수 있다. 반드시 게시글 상세 URL로 들어가 본문이 끝까지 펼쳐졌는지 확인한다.

## 수집 절차

1. 브라우저에서 BAND에 로그인한다.
2. 수집할 판매처의 목록 페이지를 연다.
3. 최근 7일 이내 게시글 중 가격표 게시글을 찾는다.
4. 게시글 상세 URL을 새로 연다.
5. 본문에 `더보기` 또는 접힌 영역이 있으면 모두 펼친다.
6. 가격표가 텍스트인지 이미지인지 확인한다.
7. 텍스트 가격표는 원문을 읽고 판매처, 게시글 URL, 게시 시각, 원문 본문을 `raw_posts`에 남긴다.
8. 이미지 가격표는 눈으로 읽을 수 있으면 수동 전사하고, 전사가 어렵거나 가격 정보가 불완전하면 `skipped`로 두고 이유를 `parse_error`에 남긴다.
9. 파싱한 항목은 어종, 국가명, 세부 원산지, 자연산/양식, 상태, 크기, 단위, kg당 가격, 품절/이벤트/반반 가능 여부를 분리해 `item_snapshots`에 입력한다.
10. 입력 후 검수 페이지에서 원문, 파싱 상태, skip 이유, 시세판 표시를 확인한다.

## 필드 분리 원칙

어종명에는 품목명만 남긴다. 예를 들어 `일본산 가리비`는 어종 `가리비`, 국가명 `일본산`으로 분리하고, `자연산 감성돔`은 어종 `감성돔`, 생산 구분 `자연산`, 세부 원산지 `자연산`으로 분리한다.

상태값은 가능한 한 별도 필드로 나눈다.

- DB 호환 원산지: 기존 `origin`은 국가명 값으로 유지한다.
- 국가명: `origin_country`에 `국내산`, `일본산`, `중국산`, `노르웨이`, `러시아` 등 국가 단위 값을 넣는다.
- 세부 원산지: `origin_detail`에 `낚시바리`, `자연산`, `제주산` 같은 지역산, `양식` 중 하나만 넣는다. 우선순위는 `낚시바리 > 자연산 > 지역산 > 양식`이다.
- 생산 구분: `자연산`, `양식`
- 신선 상태: `활어`, `선어`, `찍어바리`, `꼬물이`
- 크기: `1.3~1.5kg`, `2kg+` 등
- 등급/설명: `A급`, `상급`, `상태최강`, `예약판매` 등. `낚시바리`는 원산지 세부값으로 우선 기록하고, 별도 등급 의미가 있을 때만 등급/설명에도 남긴다.
- 판매 상태: `품절`, `이벤트`
- 반반 가능 여부: `half_available`에 별도 boolean 값으로 남긴다.

예시:

- `광어, 제주산, 양식` → `canonical_name/display_name = 광어`, `origin = 국내산`, `origin_country = 국내산`, `origin_detail = 제주산`, `production_type = 양식`
- `광어, 국내산, 낚시바리, 자연산` → `canonical_name/display_name = 광어`, `origin = 국내산`, `origin_country = 국내산`, `origin_detail = 낚시바리`, `production_type = 자연산`

같은 판매처의 같은 어종에 여러 가격 조건이 있으면 국가명과 세부 원산지까지 비교해 행을 나눈다. 화면에서는 `품목명` 또는 `품목명` 줄바꿈 `(세부 원산지)`로 보이며, 국가만 있고 세부 원산지가 없으면 괄호를 표시하지 않는다. 중량은 가격 오른쪽에 표시하고 태그에는 넣지 않는다. `half_available`이 참이면 중량 오른쪽에 `(반반)`으로 표시하고 태그에는 넣지 않는다.

## Skip 사유 기록

수집했지만 시세 항목으로 넣지 못한 게시글은 반드시 이유를 남긴다. 이미 수집된 원문도 예외 없이 `parse_error`가 검수 페이지에 보여야 한다.

권장 사유 형식은 짧고 구체적으로 적는다.

- `manual: image-only price post; readable text not captured`
- `manual: not a price post`
- `manual: price image too blurry to transcribe`
- `manual: older than 7-day collection window`
- `manual: duplicate post already captured`
- `manual: sold-out notice only`

## 보안과 마스킹

BAND 로그인 쿠키, 계정 정보, 전화번호, 계좌번호는 문서나 DB에 그대로 남기지 않는다. 원문 저장 전에 기존 마스킹 로직으로 연락처와 계좌성 문자열을 숨긴다. 브라우저에서 직접 읽은 내용도 API 수집 원문과 동일하게 마스킹된 원문만 검수 화면에 노출한다.

## Page를 이전에 못 읽은 이유

기존 `band_api` 소스는 BAND 공식 Open API의 밴드 목록과 게시글 API를 기준으로 동작했다. 그러나 `band.us/page/...` 게시글은 공식 밴드 키 기반 게시글 목록에 잡히지 않았고, 공개 HTML에도 전체 가격 본문이 들어있지 않았다. 따라서 Page 게시글은 브라우저에서 로그인된 상태로 상세 페이지를 열고, 사람이 펼쳐진 본문을 확인해 직접 수집해야 한다.

## 운영 체크리스트

- 게시글 상세 URL을 저장했는가?
- 최근 7일 범위에 들어오는 게시글인가?
- `더보기`를 펼쳐 전체 본문을 봤는가?
- 이미지 가격표라면 수동 전사 가능 여부를 판단했는가?
- 어종명에서 국가명/세부 원산지/자연산/상태/크기를 분리했는가?
- `origin`과 `origin_country`가 국가명으로 유지되고, `origin_detail`이 우선순위에 맞게 하나만 기록됐는가?
- 같은 어종의 다른 조건은 별도 행으로 입력했는가?
- 중량과 반반 가능 여부가 태그가 아니라 가격 오른쪽 보조 표기로 보이는가?
- skip된 원문에 구체적인 이유를 남겼는가?
- 검수 페이지와 시세판에서 실제 표시를 확인했는가?
